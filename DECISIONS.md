## 🟢 CORE — build the product on these

### Modal sandbox/function backend (primary alt-provider for arbitrary code)  — 8/10
*Layer: `product-strategy` · works: **yes** · ToS risk: **low** · fragility: **low***

Wrap modal.Sandbox.create() for arbitrary/agent-generated code and modal.Function(gpu=...) for structured workloads, authenticated by a token id/secret. Spawn a GPU sandbox, run commands via sb.exec(), mount modal.Volume for file I/O, stream stdout/stderr, and invoke deployed functions externally via modal.Function.from_name(). Maps cleanly onto the provider-abstraction submit/status/logs/fetch interface.

**Verdict:** Technically every concrete claim in the proposal is accurate and verified against Modal's own 2025-2026 docs: Sandbox.create(), sb.exec() with streaming stdout/stderr iterables, GPU types T4 through B200, 5-min default / 24h max timeout, token-id/secret auth, modal.Volume I/O, and external invocation via from_name(). Modal Sandboxes are gVisor-isolated and purpose-built for untrusted/agent-generated code, which is an EXACT fit for the 'arbitrary/agent code' use case. Crucially, unlike a browser-automation-of-Colab approach, this has essentially ZERO bot-detection, account-ban, or UI-scraping fragility — it is a sanctioned first-party API. That is why fragility and ToS risk are low and the viability score is high. The one big asterisk is NOT technical: Modal is not Colab. It abandons the literal 'control Google Colab' goal in favor of replacing the backend. As a way to 'run ML/GPU code without the website' it is the single most robust and Python-native option reviewed. As 'programmatic control of Google Colab Pro' it is non-compliant by definition. Modal even offers Notebooks (GPU Jupyter that runs .ipynb, boots in seconds, built on Sandboxes), which partially neutralizes the proposal's own '.ipynb-not-native' con. Cost and runaway-agent spend are the real things to engineer around, not breakage.

**Failure modes:**
- STRATEGIC MISMATCH (the dominant risk): the parent goal is 'programmatic control of Google Colab Pro.' Modal is a different cloud entirely and touches no Colab/Google infrastructure. If 'Colab' is a hard requirement (e.g. user wants their existing Colab Pro subscription, Google Drive mounts, free-tier credits, or Colab-specific notebooks), Modal does not satisfy it at all. It only satisfies the *underlying* goal ('run ML/GPU code without a browser'). Whoever owns the spec must explicitly accept that this is a substitute-the-backend pivot, not a control-Colab path.
- COST, not breakage, is the failure mode: Modal is pay-per-GPU-second with no free GPU tier beyond a $30/mo Starter credit. Verified rates ~ H100 $3.95/hr, H200 $4.54/hr, B200 $6.25/hr. An always-on or agent-spammed sandbox can burn the $30 credit in hours; for an autonomous agent generating code in a loop, runaway spend is a real operational risk (mitigable with timeouts/spend caps but must be designed in).
- Sandbox default timeout is 5 min (confirmed), max 24h. Long fine-tunes need explicit timeout config and Filesystem Snapshots to span >24h — an extra moving part the abstraction must handle, and a footgun if forgotten.
- Image build / cold-start latency on first run of a new image; mitigable with pre-built images but adds a build/caching layer to the package.
- Starter plan caps: 100 containers, 10 concurrent GPUs (verified). Fine for a dev/agent, but the 'scale to 50k sessions' marketing only applies on paid tiers and after a sales conversation — do not design assuming unlimited concurrency on free tier.
- Proprietary control plane and SDK: vendor lock-in. Modal can change SDK semantics or pricing; the abstraction layer mitigates lock-in but you inherit Modal's roadmap and any API churn (less churn risk than scraping Colab, but non-zero).
- Volume consistency/concurrency semantics: modal.Volume is not a POSIX filesystem with strong concurrent-write guarantees; concurrent sandboxes writing the same volume need care.

### OS-keychain secret storage with per-account keying  — 7/10
*Layer: `authentication` · works: **yes** · ToS risk: **low** · fragility: **low***

Store all credentials (OAuth refresh tokens, cookie blobs, API keys) in the OS keychain via Python keyring (macOS Keychain backend), using one service name plus per-account-email usernames to support multiple Google accounts cleanly. Avoids the plaintext token.json files that tools like xeodou/colab-cli currently use.

**Verdict:** Technically sound and the lowest-risk piece of the whole system: `keyring` 25.7.0 (released Nov 16, 2025) is actively maintained, the macOS Keychain backend is stable, and it is strictly better than committing token.json to disk. But two things must be honest: (1) its marquee security benefit is overstated for the actual deployment — on macOS, keyring secrets are readable WITHOUT a prompt by any Python process run as the user from the same binary (keyring docs / jaraco/keyring #457), and an unattended agent will 'always allow' anyway, so the upgrade over token.json on a single-user dev Mac is modest (no plaintext on disk, no git leaks, ACL-gateable) rather than transformative; and (2) it explicitly authenticates nothing and absorbs none of the product's actual risk, which lives entirely in the auth/execution layers (Colab ToS, bot detection, token expiry, keep-alive). Watch the ~4KB Keychain item soft-limit for cookie blobs (chunk them), the first-access/binary-change prompt deadlock for headless runs, and the non-portability that forces a second backend for any non-Mac target.

**Failure modes:**
- macOS Keychain generic-password items have a ~4KB SOFT limit (16MB hard, but in-between risks securityd crashes / keychain errors per documented behavior). OAuth refresh tokens and API keys are tiny and fine, but full Google auth COOKIE BLOBS frequently exceed 4KB, causing intermittent write/read errors unless chunked across multiple items.
- The headline 'encrypted at rest' security benefit is largely illusory for this use case: keyring docs and jaraco/keyring issue #457 (closed) confirm that on macOS, secrets are accessible WITHOUT a password prompt to ANY script run from the same Python executable, and venv isolation does NOT help (venvs symlink the same python binary). For an unattended AI agent you will grant 'always allow', so any local Python process running as the user can read every credential anyway.
- The con about 'keychain prompts interrupting unattended runs' is real on first access / after a binary or code-signature change (e.g. upgrading Python, switching from system python to a packaged/PyInstaller binary, CI runners), which can deadlock a headless agent waiting on an invisible GUI prompt. Mitigation (always-allow / -T ACL) re-introduces the security weakness above.
- Pure storage layer: it carries ZERO authentication, ToS, bot-detection, or Colab-fragility risk itself, but it also solves none of those. All real product risk lives in the auth/execution layers it feeds (Colab-VS-Code-extension client-id OAuth, cookie reuse, 60s keep-alive automation that Colab's ToS explicitly discourages for managed runtimes).
- Credentials it stores are themselves perishable in ways storage cannot fix: Google refresh tokens expire at 7 days if the OAuth consent screen is in 'Testing' status, after 6 months idle, on password change (Gmail-scope tokens), or when per-client/per-user token limits are exceeded — silently breaking the stored secret regardless of how securely it is kept.
- Keychain is not portable/syncable for headless server or Linux deployment; the macOS-native backend only works on the dev Mac (macOS 11+, Python 3.8.7+ universal2). Cross-platform 'production-quality package' goal forces a second backend (SecretService/Windows Credential Manager/encrypted file), so the macOS Keychain win does not generalize.
- iCloud Keychain sync (if Keychain Access is iCloud-enabled) could replicate sensitive long-lived Google credentials across the user's other Apple devices unexpectedly — a quiet attack-surface expansion.

### Provider-abstraction backend interface (submit/status/logs/fetch/cancel)  — 7/10
*Layer: `product-strategy` · works: **partially** · ToS risk: **medium** · fragility: **medium***

Define a common remote-execution interface — submit(code_or_image, gpu, env, files), status(handle), logs(handle, stream?), fetch_outputs(handle), cancel(handle) — and implement it across pluggable backends (Colab consumer, Colab Enterprise, and third-party GPU providers). Feature-detect capabilities like real-time log streaming (Modal/HF Jobs/RunPod/SageMaker/Vertex support it; Kaggle is poll-then-fetch only) and interactive vs batch execution. Positions provider portability as a first-class feature given Colab's inherent fragility.

**Verdict:** The abstraction itself is sound and arguably the single best strategic decision in this product. Verified against primary sources: the non-Colab backends are real, stable, ToS-blessed Jobs APIs that map cleanly onto submit/status/logs/fetch/cancel — Modal (SDK, streaming logs, GPU), HF Jobs (run/logs/cancel/ps/inspect + scheduled jobs), RunPod (REST API + Python SDK, real-time logs, pod lifecycle), Vertex AI / Colab Enterprise (CustomJob submit/get/cancel + Cloud Logging), Kaggle (push/status/output/pull, genuinely poll-then-fetch, no live logs). The capability matrix in the proposal is factually accurate. The catch: the one backend the whole product is named after — Colab consumer/Pro — remains the fragile exception. There is no official consumer API; the official googlecolab/colab-mcp (2026-03-17, v1.0.2) is a LOCAL browser-bridge that requires an open, logged-in Colab tab and offers no headless mode, so within this abstraction the Colab node is a degraded, human-supervised, capability-reduced backend, and it carries the only meaningful ToS/ban exposure. That is, paradoxically, exactly what this pattern is for: isolating the brittle provider behind a stable interface and letting users route around it. The abstraction does not fix Colab's fragility, but it correctly contains it and makes the product survivable even if the Colab path degrades or gets shut off. Lower confidence on the long-term stability of the Colab MCP bridge specifically (Google's support bandwidth is explicitly limited, internals change often).

**Failure modes:**
- Load-bearing weakness is the Colab-consumer backend: there is NO official consumer Colab API. The official googlecolab/colab-mcp (released 2026-03-17, v1.0.2) is explicitly 'a bridge to a Colab session in the browser' — must run locally, requires an OPEN, logged-in Colab browser tab, and has no headless/server mode. So the Colab node can only ever be a human-supervised, browser-tethered, degraded backend inside the abstraction — it cannot meet the same submit/poll-unattended contract as the others.
- Lowest-common-denominator interface: to keep the contract uniform, real provider features get hidden (Modal sandboxes/snapshots, Vertex hyperparameter tuning, RunPod pod lifecycle, HF Jobs scheduled cron). The 'capability feature-detection' escape hatch mitigates this but pushes complexity onto every caller and erodes the 'one stable interface' promise.
- Capability parity is genuinely impossible and the design admits it: Kaggle is poll-then-fetch only (kernels push/status/output/pull — no live logs, verified), Colab consumer has no log-streaming contract and undocumented 30-90 min idle/disconnect + session recycling. Callers must branch on capabilities, which is where bugs and surprising behavior accumulate.
- ToS/ban risk is concentrated on the Colab path: consumer ToS forbids accessing Colab 'other than as authorized by Google' and forbids selling/renting/providing access to third parties. A product that proxies/resells Colab Pro compute to an agent (especially server-side) is squarely in that zone. The MCP server sanctions a LOCAL browser-bridge pattern only — it does not bless server-side automation or resale.
- Maintenance surface: each backend's API drifts independently (Kaggle CLI, RunPod REST v2, huggingface_hub Jobs, Vertex SDK, Modal client). The abstraction must track 5-6 moving SDKs plus the most volatile one (Colab MCP, no external PRs accepted, discussions-only issue tracker — low support bandwidth signal).
- Branding/expectation mismatch: the product goal is 'programmatic control of Google Colab Pro,' but the durable value of THIS layer lives in the non-Colab backends (Modal/HF/RunPod/Vertex), which are real supported Jobs APIs. The Colab headline is the fragile part; the abstraction quietly relocates value away from it.

### Wrap/extend the official google-colab-cli and colab-mcp  — 6/10
*Layer: `product-strategy` · works: **partially** · ToS risk: **low** · fragility: **high***

Vendor or shell out to the Apache-2.0 google-colab-cli (the now-recommended official headless CLI) and reuse/extend colab-mcp for agent integrations, authenticating via the CLI's OAuth2 loopback or ADC. Build the package as a higher-level layer over Google's own maintained tools rather than reverse-engineered endpoints.

**Verdict:** This is the LOWEST-ToS-risk path available and the only genuinely sanctioned one: googlecolab/google-colab-cli and googlecolab/colab-mcp are real, official, Apache-2.0, Google-published (April 2026 blog endorses 'treating Colab as an automated workspace for any MCP-compatible agent'), and the Colab FAQ explicitly LIFTS the UI-bypass/remote-control prohibition for paid plans with a positive compute-unit balance — which is exactly the Colab Pro target. The CLI is purpose-built for agents (GPU/TPU sessions, run code, sync files, capture notebooks) and is the right primary surface. BUT the foundation is immature and fragile: the CLI is v0.5.x with yanked releases, Python-3.13-only, no confirmed stable JSON mode (stdout parsing risk), and Google rejects external contributions so you own all breakage. The headlessness pros are overstated — colab-mcp explicitly requires a human-opened browser session; only the newer CLI is closer to headless, and its public-OAuth-client story for arbitrary external users is unverified. Account-ban risk persists even for paying users via opaque, no-recourse abuse detection. Net: build on this because nothing else is sanctioned, but treat it as a fast-moving dependency with a hard abstraction layer, pin versions, prefer the CLI over the browser-bound MCP, and assume per-account ban exposure and frequent interface breakage rather than a stable platform.

**Failure modes:**
- Interface churn: official CLI is v0.5.x with YANKED releases on PyPI (0.5.5/0.5.6 yanked) and Python>=3.13-only. A 0.x tool changing weekly is a brittle foundation for a 'production-quality' package; CLI flags, output format, and config paths (~/.colab-cli-oauth-config.json) can break between point releases. Google explicitly rejects external contributions on both repos, so you cannot upstream fixes and must chase breakage downstream.
- No stable JSON output mode confirmed: wrapping the CLI likely means parsing human-readable stdout, which silently breaks on cosmetic output changes. This is the classic shell-out fragility the approach's own cons admit.
- OAuth public-client assumption UNVERIFIED: docs show a default oauth2 flow plus an optional --client-oauth-config, which IMPLIES a bundled client exists, but Google has not documented a public client guaranteed usable by arbitrary external end users. If Google gates, rotates, or scopes that client to first-party use (the exact 'oauth-borrowed-vscode-client' dependsOn risk), distributed external auth breaks. Could not confirm from primary sources.
- colab-mcp is NOT headless: its README/blog describe 'bridging your local agent to a Colab session in the browser' and require a client supporting notifications/tools/list_changed running locally. The MCP integration pro (Claude Code/Gemini/Windsurf) is real but tied to a human-opened browser tab, contradicting 'full programmatic control without the website.' The CLI is the more-headless path but is the newest/least proven piece.
- Headless auth friction: documented headless story is 'authenticate on a machine with a browser, then copy the token file.' Unattended/CI servers need durable refresh-token handling that is unverified; tokens expire and re-consent is interactive.
- Abuse-detection bans even for PAID users: Colab FAQ permits UI-bypass/remote-control on paid runtimes with positive compute balance, but colabtools issues #4979 and #4986 show paying users blocked for 'suspected abusive activity' with no explanation and no working appeal. Google states it will not disclose detection heuristics and 'may terminate at any time without warning.' Sustained agent-driven GPU usage is exactly the high-volume pattern that trips these heuristics, so per-account ban risk is real even on the sanctioned path.
- Maturity: Google itself frames colab-mcp as 'an early step toward more seamless integration' and routes everything to GitHub Discussions (issues/contributions effectively closed). Building a product on a feedback-stage tool means betting on roadmap stability that does not yet exist.
- Compute-unit economics/limits: COLAB_SKILL warns idle VMs burn compute units and to always stop sessions; A100/H100/TPU access on Pro is rate/quota limited and not contractually guaranteed for sustained headless workloads, capping reliability and scale.

### Runtime-proxy token + assignment lifecycle management  — 5.5/10
*Layer: `runtime-allocation` · works: **partially** · ToS risk: **medium** · fragility: **high***

Treat the short-lived runtime-proxy token (with explicit tokenExpiresInSeconds) and the assignment itself as managed, refreshable resources: re-fetch the proxy token via /tun/m/runtime-proxy-token on expiry, and re-assign on idle (~90 min free) / max-lifetime (12 h free, 24 h Pro) termination, handling HTTP 412 TooManyAssignmentsError and the quota Outcome enum (SUCCESS/DENYLISTED/QUOTA_*).

**Verdict:** The lifecycle problem this component solves is genuinely mandatory — without proxy-token refresh and re-assignment handling, kernels die silently and no long-running workload survives, so SOMETHING like this must exist in the product. The good news, which materially raised my score above the usual 'reverse-engineered Colab' floor: these are not hostile scraped endpoints. The exact paths (/tun/m/runtime-proxy-token, /tun/m/assign, /tun/m/assignments), the tokenExpiresInSeconds field, and TooManyAssignmentsError are all present in Google's OWN official, Apache-2.0-licensed colab-vscode extension (launched 2025-11-13), which connects to Pro-tier premium-GPU runtimes. And crucially for a Colab PRO target, the FAQ confirms paid users with a positive compute-unit balance have the 'bypassing the notebook UI' and 'remote control' restrictions lifted — so a single-account, paid, interactive-style client is much closer to sanctioned than the free-tier or multi-account case. The bad news is everything around durability: Google documents zero API contract or stability guarantee, calls the limits explicitly unpublished and variable, positions the extension strictly for interactive IDE use, the OAuth-for-headless story decays every 7 days under 'Testing' status, the 412 concurrency cap pushes scaling straight into the explicitly-banned multi-account/anti-abuse-circumvention territory, and I could not confirm the literal 412 binding or the SUCCESS/DENYLISTED/QUOTA_* enum from primary sources (medium confidence on those specifics; Colab internals are a moving target). Net: the SINGLE-account Pro happy path will work today and is the most defensible piece of the whole product, but it will break on Google's schedule, not yours, and it cannot legitimately scale multi-tenant.

**Failure modes:**
- Token/endpoint drift: tokenExpiresInSeconds, /tun/m/runtime-proxy-token, /tun/m/assign and the assignment schema are first-party-internal, NOT a documented public API. Google ships them via the official Apache-2.0 googlecolab/colab-vscode extension (launched 2025-11-13) but explicitly positions that extension for interactive IDE use only and publishes no API contract or stability guarantee. They can change field names, paths, headers (X-Colab-Runtime-Proxy-Token, X-Goog-Colab-Client-Agent, authuser=0), or the XSRF two-phase flow at any time, silently breaking the lifecycle manager.
- Unconfirmed schema specifics: I could verify TooManyAssignmentsError as a real custom error type and the runtime-proxy-token/tokenExpiresInSeconds fields in the extension source (via DeepWiki), but I could NOT find primary confirmation of the literal HTTP 412 status binding or the exact quota Outcome enum values (SUCCESS/DENYLISTED/QUOTA_*). Treat those specific names as plausible-but-unverified; coding hard branches against them is fragile.
- Heuristic reconnect against unpublished limits: the Colab FAQ states outright that usage limits 'can vary over time' and are deliberately unpublished, and that even paid users reverting to a zero CCU balance fall back to free-tier policy. The 90-min idle / 12h / 24h numbers are community-observed, not contractual, so re-assign timing will be guesswork and will misfire when Google tightens limits during peak demand.
- State loss on re-assignment: re-assign after idle/lifetime termination gives a fresh VM with no in-memory or local-disk state. The lifecycle manager alone does not solve this; it forces a mandatory dependency on a separate checkpoint/restore layer (Drive/GCS) or every long job silently restarts from zero.
- OAuth automation friction / token decay: the official flow requires the operator to create their OWN GCP OAuth 2.0 desktop client. An unverified/'Testing' app with sensitive scopes has refresh tokens revoked after 7 days, breaking any unattended agent weekly; moving to 'Production' requires Google OAuth verification for sensitive scopes. This is a real, recurring operational failure for a headless agent, not a one-time setup cost.
- Concurrency cap is structural, not tunable: 412 TooManyAssignmentsError reflects a per-account assignment limit. The approach's own 'cons' admit multi-tenant use needs per-user accounts. Spinning up many accounts to scale runs directly into the FAQ's explicit ban on 'using multiple accounts to work around access or resource usage restrictions' and 'containerization to circumvent anti-abuse policies' — i.e., the obvious scaling path is the prohibited path.
- Bot/abuse detection and enforcement is real: a documented case exists of a Google account suspended for non-sanctioned Colab tunneling (colab-ssh/ngrok). While direct-backend assign is closer to first-party than ngrok tunneling, a fully headless, no-browser, agent-driven client diverges from the sanctioned interactive VS Code pattern and risks denylisting (the DENYLISTED outcome the approach itself anticipates) or account action if it looks like automated resource farming.

### Standard Jupyter kernel execution over WebSocket (jupyter-kernel-client)  — 4.5/10
*Layer: `code-execution` · works: **partially** · ToS risk: **medium** · fragility: **high***

Connect to the runtimeProxyInfo.url as an ordinary Jupyter server using the datalayer jupyter-kernel-client: REST POST/GET/DELETE /api/kernels for lifecycle and ws(s)://<server>/api/kernels/{id}/channels for execution, passing the proxy token as Bearer header, X-Colab-Runtime-Proxy-Token header, AND colab-runtime-proxy-token query param. Send execute_request on the shell channel and assemble iopub messages (stream/execute_result/display_data/error/status) into {execution_count, status, outputs}.

**Verdict:** Technically the most credible execution layer of the options, because it mirrors Google's own first-party design: the official open-source colab-vscode extension uses /tun/m/assign + RuntimeProxyInfo + X-Colab-Runtime-Proxy-Token, and jupyter-kernel-client (wsclient.py uses websocket.WebSocketApp with header= and supports both DEFAULT and v1.kernel.websocket.jupyter.org subprotocols) is technically able to inject custom headers and the token. So execution-over-WebSocket WILL work in principle. But the approach's stated auth recipe is wrong in detail: the proxy token is a header-only credential distinct from the OAuth Bearer identity token, and the description omits the X-Goog-Colab-Tunnel and X-Goog-Colab-Token XSRF headers the real client sends. ToS risk is lower than feared because the FAQ's 'no UI bypass / no remote control' prohibitions apply to free runtimes and are explicitly lifted on paid plans (target is Colab Pro), but it is not zero: client-agent fingerprinting, opaque 'suspected abusive activity' bans, and emerging agent-specific blocks (issue #433) mean a headless AI-agent driver remains exposed. Everything rests on undocumented internals that Google changes at will, and even the sanctioned client is visibly flaky in 2025-2026.

**Failure modes:**
- Auth contract is mis-specified. Google's own open-source extension (googlecolab/colab-vscode, src/jupyter/assignments.ts colabProxyFetch) injects the runtime proxy token ONLY as an HTTP header (headers.set(COLAB_RUNTIME_PROXY_TOKEN_HEADER.key, token)), NOT as a Bearer token and NOT as a query param. The Authorization: Bearer header carries a SEPARATE Google OAuth identity token. The approach's 'send it three ways including Bearer' will at best be ignored and at worst collide with the real OAuth Authorization header and break the request. The proxy also expects X-Goog-Colab-Tunnel: true and an X-Goog-Colab-Token XSRF value that the approach does not mention at all.
- Kernel lifecycle is unverified. It is undocumented whether you must POST /api/kernels through the proxy or whether the VM auto-provisions a session; jupyter-kernel-client can do either, but if Colab's proxy requires a specific session_id/path/name convention or an XSRF token on POST, naive kernel creation will 403/timeout (cf. colab-vscode issue #427 'Timeout flake on GET kernels call', #604 'socket hangup on assign').
- Client fingerprinting / bot detection. The proxy reads X-Goog-Colab-Client-Agent to identify the VS Code extension. A bespoke client must either spoof 'vscode' (fragile, detectable, and a clear ToS-circumvention signal) or send its own agent string that Google can throttle/block at will. Issue #433 already shows AI-agent-driven GPU execution failing on token/auth grounds, indicating Google is actively adding agent-specific friction.
- Opaque, non-appealable anti-abuse blocks. colabtools issues #4979/#4986/#5038 show accounts hit with 'blocked due to suspected abusive activity' with no visible Google response and unclear appeal path. High-volume, headless, long-running programmatic GPU use is exactly the signature these heuristics target, even on Pro.
- Inherited transport fragility. /tun/m/assign, RuntimeProxyInfo URL format, token rotation, keep-alive (/tun/m/{id}/keep-alive every 5 min), and idle timeouts are all undocumented internals that change without notice. The approach explicitly dependsOn three other reverse-engineered layers (direct-backend-tun-assign, runtime-proxy-token-lifecycle, tun-tunnel-header-proxy), so it is only as stable as the weakest of those.
- Reliability is poor even on the sanctioned path. Current 2025-2026 colab-vscode issues show socket hangups, kernel GET timeouts, OPENSSL BAD_DECRYPT on auth calls, and activation failures. A non-Google clone gets all of this plus divergence whenever Google updates the contract.
- jupyter-kernel-client maturity risk: v0.6.0 (Mar 2025), small surface, not designed for Colab's bespoke header/XSRF/keep-alive requirements; you will be forking or monkeypatching it, eroding the 'reuse mature OSS' pro.

### Self-registered Desktop OAuth client with the colaboratory scope  — 3.5/10

For a private tool this gate is never passable, so you are permanently stuck on the 7-day treadmill.

- ToS exposure: Colab ToS v5 prohibits accessing 'Colab other than as authorized by Google' and bans sharing/reselling access. Self-registering a client to hit the private backend with the colaboratory scope is plausibly 'other than authorized', though mitigated by the fact that Google itself ships an official extension using the identical scope and backend. Account-ban risk is real but lower than overt automation/scraping.

- Auth-flow path coupling: colab-vscode's proxied flow redirects to a Google-owned endpoint (${ColabApiDomain}/vscode/redirect); a self-registered client cannot register that redirect URI. ### Papermill/nbconvert notebook-execution adapter over any kernel backend  — 6/10
*Layer: `code-execution` · works: **partially** · ToS risk: **low** · fragility: **low***

Layer a notebook-execution adapter (papermill/nbconvert-style) on top of any kernel-capable backend so the package can run actual .ipynb notebooks cell-by-cell with parameter injection and output capture, regardless of whether the underlying backend exposes a native 'run notebook' primitive. Bridges the gap between 'run arbitrary code' backends (Modal/HF Jobs) and 'run a notebook' semantics (Kaggle/Vertex).

**Verdict:** Technically sound and low-risk as an adapter — nbclient explicitly supports custom/remote kernel managers (GatewayKernelManager), and papermill/nbclient are mature and actively maintained (papermill 2.7.0 Feb 2026, nbclient 0.10.4 Dec 2025), so the 'another point of failure' con is overstated. The ToS/ban/fragility risk this review was asked to hunt for does NOT live in this layer; it lives entirely in the upstream websocket-exec dependency. The real problem is strategic fit: batch full-notebook execution is the wrong granularity for an AI agent that wants interactive cell-by-cell control, and rich/widget outputs round-trip lossily. This is a genuinely useful capability for parameterized, reproducible batch runs and for letting code-only backends (Modal, HF Jobs) accept .ipynb workloads — but it should not be the spine of an interactive Colab-control product.

**Failure modes:**
- Wrong granularity for the agent use case: papermill/nbclient run a whole notebook as a batch and discard interactive/stateful REPL semantics. An AI agent iterating cell-by-cell and reading intermediate kernel state is exactly what this layer throws away — confirmed by nbclient/papermill design (full-notebook execute, output written back to .ipynb).
- Misnomer / leaky abstraction: papermill's higher-level operator/CLI does NOT support externally-managed remote kernels (open enhancement Apache Airflow #45866). You'd actually build on nbclient.NotebookClient with a custom AsyncKernelManager / kernel_manager_class (GatewayKernelManager pattern), not papermill itself. The 'papermill' label overstates what you reuse — only parameter injection (parameters→injected-parameters cell tag) and .ipynb output capture genuinely transfer.
- Rich/widget output does not round-trip: ipywidgets render only as text reprs because there is no browser during execution (papermill #400); high-frequency output (tqdm) must be throttled or floods/garbles capture. Reproducibility claim is weaker than advertised for any notebook using interactive display.
- Reliability over a high-latency remote Colab kernel: nbclient relies on per-cell timeouts and a persistent websocket; a dropped websocket can leave the UI showing 'kernel available' while execution silently stalls (jupyter/notebook #4757). Long GPU training cells amplify timeout/disconnect fragility, all of which this layer inherits, not fixes.
- Dependency, not differentiator: ALL Colab bot-detection, account-ban, and auth-friction risk lives in the upstream jupyter-kernel-websocket-exec / Colab-reach layer, not here. This layer cannot mitigate any of it. Meanwhile Google shipped an official googlecolab/colab-mcp (Mar 2026) that bridges to a browser Colab session — a sanctioned path that competes with rolling your own websocket exec for the Colab case specifically.
- Extra moving part with low payoff if the product is Colab-only: the adapter's main value (unifying notebook semantics across many backends) only pays off if you actually run multiple heterogeneous backends; for a Colab-Pro-focused product it is largely overhead on top of the kernel layer.

- Architectural mismatch with the stated goal. The product explicitly targets 'clean headless servers,' but this approach requires a browser installed and logged in on the same machine. The approach's own cons concede it is 'not viable for clean headless servers.'
- Tool fragility: browser_cookie3 has recurring 2025 decryption-failure issues (e.g. #224 'unable to decrypt cookies', #221 Edge key failure) and is tested against single browser versions. Chrome cookie-encryption internals (v10/v11/v20, ABE expansion) change faster than the library tracks. App-Bound Encryption is currently Windows-only, but Google has stated intent to expand cookie protections beyond Windows.
- Even when cookies are valid, Colab's backend is undocumented and changes without notice; SAPISIDHASH transport and Colab's WebSocket/runtime-assignment APIs can break independently of the auth layer, compounding fragility.

### In-page kernel driving via output-iframe context (not recommended)  — 1/10
*Layer: `code-execution` · works: **no** · ToS risk: **prohibited** · fragility: **high***

Attempt to execute code by injecting JavaScript into a Colab cell-output iframe to reach google.colab.kernel.invokeFunction / the kernel WS, inheriting Colab's cookies and the x-colab-tunnel header same-origin. Catalogued for completeness; google.colab.kernel is exposed only inside sandboxed per-cell output iframes, not the top-level page.

**Verdict:** Confirmed non-starter, and the approach's own description and cons are accurate. The kernel API (google.colab.kernel.invokeFunction) only exists inside Colab's own sandboxed per-cell output iframes, which are cross-origin (sandbox=allow-scripts, no allow-same-origin, origin null) and which Colab generates and controls; you cannot deterministically inject into one from a driven top-level page, and the one iframe primitive a caller could create (serve_kernel_port_as_iframe) has window.google undefined and is itself a documented-but-broken, unfixed bug. Even if you cleared the context wall, you would be reverse-engineering an undocumented socket.io + comms message schema with no stability guarantee against an internal surface that already breaks its own public APIs without notice. On top of the technical impossibility, it squarely violates the Colab Paid ToS clause forbidding access 'other than by means authorized by Google' and matches the automation/tunneling patterns that have produced real Colab runtime bans (issue #4979) and multi-month account suspensions. The 'inherits cookies and tunnel header same-origin' pro is mostly false because the sandbox boundary breaks same-origin inheritance. There is no path here to a production-quality, sustainable package for an AI agent or developer.

**Failure modes:**
- Wrong execution context by design: google.colab.kernel lives only inside Colab's own per-cell output iframes, not the top-level page JS namespace (confirmed by colabtools maintainer-built architecture and the Matteo Ferla writeup: 'available in the cells' outputs, but not in the namespace'). You cannot just inject JS into the page and call invokeFunction.
- The one user-facing iframe entry point is broken/unusable for this: serve_kernel_port_as_iframe iframes have window.google === undefined (colabtools issue #3253, labeled bug+triaged, no fix), so the kernel API is not reachable from the iframe context an external caller could create.
- Cross-origin sandbox wall: Colab output iframes use sandbox=allow-scripts without allow-same-origin, so their origin is null and the parent page cannot script into them and they cannot script the parent. An automation injecting into the top page cannot reach the kernel-bearing iframe's context, and the kernel iframe cannot reach Colab cookies/top-level same-origin. The 'inherits cookies + x-colab-tunnel same-origin' premise is largely false across the sandbox boundary.
- Undocumented private protocol: invokeFunction rides Colab's socket.io multiplexing plus a private comms/message schema (no published spec). Building a robust execute/reply loop against this is reverse-engineering a moving target with zero stability guarantee.
- Colab changes these internals at will and breaks even its own documented APIs: serve_kernel_port_as_iframe broke (colabtools issue #4037, Oct 2023) and sat unfixed. Any in-page injection would break on UI/runtime redeploys with no notice.
- ToS violation: Colab Paid ToS prohibits 'access the Paid Service other than by means authorized by Google.' Injecting JS into internal iframes to drive the kernel is textbook unauthorized access.
- Account/runtime ban risk: Google issues Colab-specific blocks for 'suspected abusive activity' with opaque criteria and ineffective appeals (colabtools issue #4979), and tunneling/automation patterns have caused multi-month suspensions (documented colab-ssh + ngrok case). Programmatic non-interactive driving is exactly the fingerprint they flag.
- Hard dependency on headless-browser UI automation (its dependsOn) inherits all of that approach's bot-detection and fragility risk on top of this layer's problems.
