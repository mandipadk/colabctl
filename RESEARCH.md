# Research Dossiers

Sourced findings from 7 parallel research tracks. Confidence is the researcher's own; Colab internals drift, so treat protocol specifics as *as-of 2026-05-31*.

## Colab internals & the Jupyter wire protocol  *(confidence: medium)*

Colab's frontend speaks standard Jupyter over WebSocket, but it is NOT exposed directly to clients. The kernel lives on a Google-provisioned VM that is only reachable through an authenticated reverse proxy (a "runtime proxy") fronted by colab.research.google.com. The definitive, current (2026) blueprint for non-browser control is Google's own open-source repo googlecolab/colab-mcp. Its git history contains a now-removed but fully working "Runtime Mode" (commit a9a4203d, src/colab_mcp/client.py + runtime.py + auth.py) that does exactly what this track asks: OAuth2-authenticate, call the Colab "/tun/m/assign" provisioning endpoint to allocate a GPU/TPU VM, receive a runtimeProxyInfo {url, token}, then connect to that URL as an ordinary Jupyter server using the datalayer jupyter-kernel-client over the standard /api/kernels/{id}/channels WebSocket to run execute_request and stream iopub outputs. An external client CAN replicate this concretely. The two hard, fragile dependencies are: (1) obtaining a valid auth token for the OAuth scope https://www.googleapis.com/auth/colaboratory (colab-mcp ships a public installed-app client_id/secret), and (2) the assign endpoint shape, accelerator enum values, and runtime-proxy token mechanics, which are undocumented and change. The current colab-mcp HEAD deliberately abandoned this direct approach in favor of a browser session-proxy (the browser does the privileged backend talk), which is a strong signal that direct backend access is brittle and possibly discouraged.

### Key findings
- DEFINITIVE PRIMARY SOURCE: googlecolab/colab-mcp (Apache-2.0, copyright 2026, v1.0.1) is Google's own MCP server for driving Colab. Its current HEAD only does browser session-proxy, but git commit a9a4203d ('Add tools for interacting with Colab VMs and Jupyter kernels', 2026-02-09) added src/colab_mcp/client.py, runtime.py, auth.py implementing FULL direct backend control. Commit 47b40efb (2026-03-24) removed them as 'not needed in this project (coming later elsewhere)'. These files are the ground truth for this track.
- VM ALLOCATION FLOW (from client.py ColabClient): assign(notebook_hash, variant, accelerator) does a two-step GET-then-POST to the endpoint /tun/m/assign on https://colab.research.google.com. GET returns a GetAssignmentResponse containing an XSRF token (fields acc, nbh, token, variant); the POST to the same URL carries header X-Goog-Colab-Token: <that token> and returns a PostAssignmentResponse. Query param nbh = the notebook hash, a UUID transformed to web-safe base64 (hyphens->underscores, padded to 44 chars with '.'). authuser=0 is appended for colab.research.google.com hosts.
- PROVISIONING RESPONSE (PostAssignmentResponse): fields are accelerator (T4/L4/A100/V2-8/V5E-1/V6E-1/NONE), endpoint (the VM id string), fit (=idle_timeout_sec), machineShape (STANDARD=0 / HIGH_RAM=1), runtimeProxyInfo {token, tokenExpiresInSeconds, url}, sub (SubscriptionState), subTier (SubscriptionTier NONE/PRO/PRO_PLUS), variant (DEFAULT=0/GPU=1/TPU=2). HTTP 412 on POST => TooManyAssignmentsError. The runtimeProxyInfo.url + token IS the reachable Jupyter server for that VM.
- RUNTIME TYPE / GPU REQUEST: requested via query params on the assign URL: variant in {DEFAULT, GPU, TPU} and accelerator in {NONE, T4, L4, A100, V2-8, V5E-1, V6E-1} (the V* values are TPU generations). So an external client requests a GPU runtime by calling /tun/m/assign?nbh=<hash>&variant=GPU&accelerator=T4 (or A100, etc.). The server returns the accelerator it actually granted plus quota outcome (Outcome enum: SUCCESS, DENYLISTED, QUOTA_DENIED_REQUESTED_VARIANTS, QUOTA_EXCEEDED_USAGE_TIME).
- KERNEL CONNECTION (from runtime.py): colab-mcp connects to runtimeProxyInfo.url with the datalayer jupyter-kernel-client as if it were a plain Jupyter server: KernelClient(server_url=url, token=token, client_kwargs={subprotocol: DEFAULT, extra_params: {'colab-runtime-proxy-token': token}}, headers={'X-Colab-Client-Agent':'colab-mcp','X-Colab-Runtime-Proxy-Token': token}). So the runtime-proxy token is passed BOTH as an HTTP/WS header AND as a 'colab-runtime-proxy-token' query param on the channels websocket. It then calls kernel.execute(code).
- JUPYTER WIRE PROTOCOL: jupyter-kernel-client manager.py uses standard REST: POST /api/kernels {name,path} to create, GET /api/kernels to list, GET/DELETE /api/kernels/{id}, POST .../restart, .../interrupt; Authorization: Bearer <token>. The kernel WebSocket is ws(s)://<server>/api/kernels/{kernel_id}/channels?session_id=<id>&token=<token> (plus colab's extra colab-runtime-proxy-token param). Subprotocol is either DEFAULT (no subprotocol; JSON messages with channel/header/parent_header/metadata/content) or 'v1.kernel.websocket.jupyter.org' (binary, 64-bit LE offsets). colab-mcp uses DEFAULT.
- CELL EXECUTION + OUTPUT STREAMING: client sends an execute_request on the shell channel (header.msg_type='execute_request', unique msg_id, content={code, silent, store_history, user_expressions, allow_stdin, stop_on_error}). Results stream back on iopub: status (execution_state busy/idle), execute_input, stream (name=stdout/stderr,text), execute_result (data,metadata,execution_count), display_data (data,metadata,transient), update_display_data (matched by display_id), clear_output, error (ename,evalue,traceback). jupyter-kernel-client assembles these into {execution_count, status, outputs:[...]}. This is exactly Colab's cell model.
- AUTH (from auth.py): uses Google OAuth2 InstalledAppFlow with a HARD-CODED public client_id 366568267421-7o7krvn9105p1ba0p3ahnnf8upt09m7h.apps.googleusercontent.com and client_secret GOCSPX-... Scopes: openid, userinfo.profile, userinfo.email, and crucially https://www.googleapis.com/auth/colaboratory. Refresh token cached in ~/.netrc (machine colab-mcp). Requests use a google.auth AuthorizedSession (Authorization: Bearer <access_token>). There is a separate Google API host colab.pa.googleapis.com (v1/user-info) vs the tunnel host colab.research.google.com (/tun/m/*).
- THE /tun/ REVERSE PROXY: confirmed by both colab-mcp (TUN_ENDPOINT='/tun/m', endpoints /tun/m/assign, /tun/m/assignments, /tun/m/unassign/{endpoint}, /tun/m/ccu-info) and the DagsHub reverse-engineering article (/tun/m/<id>/api/contents, /tun/m/<id>/files, /tun/m/<id>/_proxy/{port}). DagsHub also documents that Colab refuses to proxy unless the request carries header 'x-colab-tunnel: Google' (anti-XSS), and that the in-VM Jupyter listens on port 9000 bound to the private IP, fronted by a node app /datalab/web/app.js and a colab-fileshim on localhost:3453. Note DagsHub's 'socket.io' detail is likely outdated; current path is /api/kernels/{id}/channels.
- XSSI + ENVELOPE QUIRKS: Colab JSON responses are prefixed with the XSSI guard )]}'\n which must be stripped. Required headers on tunnel requests: Accept: application/json, X-Goog-Colab-Client-Agent: python-colab-client, and for mutating calls X-Goog-Colab-Token (XSRF). There is also a Prod vs Sandbox split (colab.sandbox.google.com / staging-colab.sandbox.googleapis.com).
- IDLE/DISCONNECT BEHAVIOR: free tier ~90 min idle timeout and 12 h max VM lifetime (Pro 24 h); these are deliberately unpublished and vary (Colab FAQ). The provisioning response carries a per-assignment idle_timeout (the 'fit' field) and the runtime-proxy token has tokenExpiresInSeconds, so a long-running external client must handle proxy-token refresh and re-assign on idle/lifetime termination.
- LOCAL RUNTIME ('Connect to a local runtime') is a SEPARATE mechanism and the only officially sanctioned non-Google-VM path: install jupyter_http_over_ws (archived 2023-09-27), run jupyter notebook --NotebookApp.allow_origin='https://colab.research.google.com' --port=8888 --NotebookApp.port_retries=0 --NotebookApp.allow_credentials=True, paste the printed backend URL into Colab. The extension adds a /http_over_websocket endpoint that tunnels HTTP requests as JSON WS messages (required keys: path, method, message_id, body|body_base64; responses chunk back {message_id,status,statusText,headers,data(base64),done}); HANDLER_VERSION 0.0.7, min_version negotiation, _xsrf cookie forwarding for POST/PUT/PATCH/DELETE.

### Candidate approaches surfaced
- **Direct backend API client (replicate colab-mcp Runtime Mode)** — Port the removed colab-mcp client.py/auth.py/runtime.py: OAuth2 with the colaboratory scope to get a Bearer token; GET then POST /tun/m/assign?nbh=<websafe-b64-uuid>&variant=GPU&accelerator=T4 on colab.research.google.com (strip XSSI prefix, send X-Goog-Colab-Token XSRF on POST); read runtimeProxyInfo{url,token}; connect to that url as a standard Jupyter server via /api/kernels REST + /api/kernels/{id}/channels WebSocket, passing the proxy token as both Bearer header, X-Colab-Runtime-Proxy-Token header, and colab-runtime-proxy-token query param; send execute_request, parse iopub. Call /tun/m/unassign/{endpoint} to release.
- **Browser session-proxy bridge (current colab-mcp HEAD model)** — Run a local WebSocket server (subprotocol 'mcp', single connection, Origin restricted to colab.research.google.com / colab.google.com, Bearer/access_token auth). Open the user's browser to https://colab.research.google.com/notebooks/empty.ipynb#mcpProxyToken=<token>&mcpProxyPort=<port>. The Colab frontend JS connects back to the local server and the browser performs all privileged backend calls (assign, kernel WS) using the user's real logged-in session. The external client drives the notebook by sending tool/JSON-RPC messages through that bridge.
- **Local-runtime via jupyter_http_over_ws (sanctioned, but no Google GPU)** — Run your OWN Jupyter server (on your machine or any GPU box) with jupyter_http_over_ws and --allow_origin=https://colab.research.google.com; Colab connects to it over the documented /http_over_websocket JSON tunnel. For pure programmatic control you can skip Colab entirely and talk to your Jupyter server directly via /api/kernels + channels WebSocket.
- **Headless-browser automation of colab.research.google.com** — Drive the real Colab web UI with Playwright/Puppeteer using stored Google auth cookies: click Connect, set runtime type, type into cells, read rendered outputs, intercept the kernel WebSocket frames via CDP.

### Risks
- Terms-of-Service / abuse-detection risk: Colab FAQ states usage limits are intentionally unpublished and variable; automated non-browser GPU allocation can be throttled, denylisted (Outcome.DENYLISTED), or treated as abuse. The fact that Google itself pulled the direct-API code from colab-mcp and shipped a browser proxy instead strongly implies direct programmatic VM assignment is not a supported/sanctioned path.
- Fragility of undocumented internals: /tun/m/assign, the PostAssignmentResponse field aliases (fit, sub, subTier, machineShape, runtimeProxyInfo), the accelerator/variant enums, the XSSI prefix, and the X-Goog-Colab-Token XSRF flow are all undocumented and have changed historically (DagsHub's older socket.io path is already stale). Expect breakage; pin to colab-mcp commits and re-verify frequently.
- Auth fragility: the hard-coded public OAuth client_id/secret (366568267421-...apps.googleusercontent.com) in colab-mcp/auth.py can be rotated/revoked by Google; the colaboratory scope may be gated. Treating a published client secret as durable is risky.
- Token/lifetime churn: runtime-proxy token has tokenExpiresInSeconds and VMs hit ~90min idle / 12h (24h Pro) limits; any production package must implement reconnect, re-assign, and state-checkpoint logic and cannot assume a stable long-lived kernel.
- Verification gap: I could read colab-mcp and jupyter-kernel-client SOURCE definitively, but I could not execute a live assign call to confirm the 2026 response shape end-to-end, nor confirm whether /tun/m/assign currently still accepts the python-colab-client agent. Confidence on the kernel/Jupyter protocol layer is high; confidence on the exact current assign/quota behavior is medium and likely to drift.
- Concurrency/quota: free and even Pro accounts allow limited concurrent assignments (HTTP 412 TooManyAssignmentsError); a multi-tenant package would need per-user Google accounts, not a shared service account.

## Existing tools & prior art  *(confidence: medium)*

Google now ships an official headless and agentic Colab toolchain; community tools mostly run servers inside the runtime behind tunnels and violate free-tier terms.

### Key findings
- Official googlecolab toolchain is the key prior art: google-colab-cli (PyPI v0.5.7, 2026-05-28, Apache-2.0, Python 3.13 plus) for headless terminal control, colab-mcp (644 stars) an MCP server, colab-vscode (345 stars) the VS Code extension.
- Reverse-engineered runtime API from the CLI source: OAuth2 loopback plus PKCE, control host colab.pa.googleapis.com, a GET then POST to colab.research.google.com tun m assign returns an endpoint and proxy token, then the Jupyter kernel runs over WebSocket execute_request through the tun m proxy with files via the Contents API, kept alive by KeepAliveAssignment. A 2020 DagsHub article confirms the same tun m proxy and x-colab-tunnel header.
- colab-mcp runs a local WebSocket server that the in-browser Colab frontend connects out to, authed by a per-instance token, exposing open_colab_browser_connection while real notebook tools are proxied from the runtime; needs tools list_changed support.
- Runtime-side google.colab message bridges kernel and frontend: colab_request on iopub, colab_reply on stdin; drive mount uses request_auth and TBE env vars; files use IPython comm channels. Website actions are just kernel messages.
- Community tools: colab-ssh (sshd via tunnels) and colabcode (code-server via ngrok) are unmaintained and violate free-tier terms; colabctl (Selenium) is broken; kora is in-runtime only; vscode-colab is the maintained tunnel successor; the xeodou Go colab-cli proves a non-Google client can drive the official runtime API by borrowing the VS Code extension OAuth credentials.

### Candidate approaches surfaced
- **Wrap or extend the official google-colab-cli and colab-mcp** — Vendor or shell out to the Apache-2.0 google-colab-cli; reuse colab-mcp for agents; auth via the CLI OAuth2 loopback or ADC.
- **Reimplement the runtime tunnel plus Jupyter protocol, clean-room** — OAuth2 loopback plus PKCE, call tun m assign for an endpoint and proxy token, keep alive via the RPC, then drive the kernel over WebSocket execute_request and the Contents API through the tunnel proxy. The xeodou Go client proves it works.
- **In-runtime server plus tunnel, legacy pattern modernized** — Inside a notebook start a server and expose via VS Code Remote Tunnels (like vscode-colab) or ngrok; the launching cell is the only browser touch.

### Risks
- Terms and ban risk dominates: Colab restricts SSH, remote desktops, and bypassing the notebook UI on free tier without a positive compute-unit balance. Prefer the official CLI.
- Internal API instability: undocumented paths change; the official CLI is v0.5.x and weeks old. Pin versions, add smoke tests.
- Auth fragility: reusing the Google VS Code OAuth client ID could be revoked; a self-registered client hits an unverified-app warning and 100-user cap.
- Headless gap: colab-mcp still needs a human-opened browser session and tools list_changed, not fully unattended.
- Source caveat: exact client IDs and accelerator codes came from repo source and one Go reimplementation, not live traffic; current as of May 2026.

- What exact OAuth client id does the CLI default to, and is it the same public client the VS Code extension and xeodou reuse?
- Can a fully headless client complete the assign POST without a browser cookie, using only a bearer plus the XSRF token?

## Terms of Service, abuse policy & ban risk  *(confidence: medium)*

Consumer Colab (Free/Pro/Pro+) and Colab Enterprise are governed by two completely different legal and policy regimes, and this distinction is the single most important finding for the project. Consumer Colab's FAQ + additional Terms (last modified 2024-05-14) + Pro Terms explicitly prohibit a large class of automation/headless patterns; some bans apply to ALL users (mining, deepfakes, DoS, password cracking, P2P/torrents, remote proxies, multi-account circumvention, containerization to evade anti-abuse) and a second set applies only to FREE (no positive compute-unit balance) managed runtimes (SSH shells, remote desktops, "bypassing the notebook UI to interact primarily via a web UI", chess training, distributed-computing workers). Google reserves the right to "reduce usage limits to zero or effectively ban" and to terminate access AND the underlying Google account without notice or refund, at sole discretion. Enforcement is real but graduated: the common outcome is opaque, dynamic "usage limits" / GPU cooldowns (hours to weeks, occasionally permanent), not account deletion; account-level bans are reserved for clearly abusive patterns (mining, 24/7 multi-account farming). Colab Enterprise is a different product: it lives inside Vertex AI / Google Cloud, is governed by the GCP Terms (not the consumer FAQ), bills via GCP, uses IAM, and EXPLICITLY supports non-interactive/scheduled/programmatic notebook execution via the Vertex AI API and runtime templates — i.e., the automation that is prohibited on free Colab is a first-class, sanctioned feature on Enterprise. Critically, in March 2026 Google shipped an official open-source MCP server (github.com/googlecolab/colab-mcp) that lets agents like Claude Code/Gemini CLI programmatically drive a Colab notebook, but its default ("session proxy") architecture bridges to a human-opened browser session rather than running headless — meaning Google's sanctioned agent path still keeps a browser/interactive frontend in the loop on consumer tiers. For a production package, the compliant high-confidence path is Colab Enterprise's Vertex AI API; consumer Colab automation (reverse-engineered API, headless browser, keep-alive auto-clickers) is contractually prohibited on free tier and carries genuine suspension risk, with all internals explicitly described by Google as undisclosed and subject to change.

### Key findings
- TWO LEGAL REGIMES. Consumer Colab (Free/Pro/Pro+) is governed by the Colab 'additional Terms of Service' (research.google.com/colaboratory/tos_v5.html, Last Modified May 14, 2024), the Colab FAQ (incorporated by reference: 'You agree to those policies and usage limitations'), and the Pro/Paid Terms (colab.research.google.com/pro/terms/v1). Colab Enterprise is governed instead by the Google Cloud Platform agreement as a Vertex AI feature. Mixing them up is the biggest risk to the project.
- ALL-USER PROHIBITIONS (apply even to Pro/Pro+ paid managed runtimes), quoted verbatim from the FAQ disallowed-activities list: 'file hosting, media serving, or other web service offerings not related to interactive compute'; 'downloading torrents or engaging in peer-to-peer file-sharing'; 'connecting to remote proxies'; 'mining cryptocurrency'; 'running denial-of-service attacks'; 'password cracking'; 'using multiple accounts to work around access or resource usage restrictions'; 'creating deepfakes'; 'employing techniques such as containerization to circumvent anti-abuse policies'.
- FREE-TIER-ONLY PROHIBITIONS (apply to 'managed Colab runtimes running free of charge, without a positive Colab compute unit balance' and 'may be terminated at any time without warning'): 'remote control such as SSH shells, remote desktops'; 'bypassing the notebook UI to interact primarily via a web UI'; 'chess training'; 'running distributed computing workers'. The web-UI clause is exactly what was used to restrict Stable Diffusion Gradio UIs on free tier in 2023.
- DIRECT AUTOMATION SIGNAL: The FAQ states 'Colab prioritizes users who are actively programming in a notebook' and 'Users in our free of charge tier commonly experience runtime terminations when attempting to bypass the notebook UI.' Non-interactive/automated batch use is precisely the behavior Colab deprioritizes and terminates on free tier.
- ENFORCEMENT POWER is broad and discretionary. Pro/Paid Terms: Google 'may, at its sole discretion, reduce usage limits to zero or effectively ban Customer from using Paid Services' and may 'terminate your access to the Paid Service and/or your Google account without notice and without refund.' Limits 'may be adjusted at the sole discretion of Google' based on 'overall capacity availability, the nature of Customer's activity ... and the activity of other users or customers.'
- LIMITS ARE DELIBERATELY OPAQUE AND DYNAMIC. FAQ: 'Colab does not publish these limits, in part because they can vary over time'; 'Colab needs to maintain the flexibility to adjust usage limits and hardware availability dynamically.' Free notebooks run 'at most 12 hours, depending on availability and your usage patterns'; 'Runtimes will time out if you are idle.' This means any reverse-engineered package built on observed limits is inherently fragile and unsupported.
- DETECTION SIGNALS (interactivity heuristic). Per detailed 2025 third-party reporting: idle timeout ~90 min across all tiers; activity is registered by browser-level user events (mouse/keyboard/clicks, manually-initiated cell runs), NOT by cell output printing, GPU utilization, outbound network requests, or auto-updating progress bars. Google also issues CAPTCHA / human-verification challenges when it 'detects suspicious patterns' — these cannot be solved by scripts, and are a clear anti-automation signal. (Lower confidence: third-party, undocumented internals.)
- REAL-WORLD BAN REALITY: The common penalty is a temporary, opaque 'usage limit' / GPU cooldown that scales with usage intensity (hours to days to weeks; occasionally the GPU becomes 'permanently unusable' on that account). Outright Google-account deletion is rare and tied to clearly abusive patterns. Reported triggers: cryptocurrency mining, running 24/7 scripts, multi-account farming to circumvent limits, and 10+ concurrent keep-alive notebooks. Moderate keep-alive use is widely reported as low-incident but is still technically against the spirit of the 'interactive' priority.
- OFFICIAL SANCTIONED AGENT PATH EXISTS (new, 2026). Google shipped an official open-source MCP server, github.com/googlecolab/colab-mcp (announced Google Developers Blog 2026-03-17, v1.0.2 on 2026-03-27), that lets MCP agents (Claude Code, Gemini CLI, Windsurf) create notebooks, run cells, install packages, and use CPU/free-T4 runtimes. ## Official Google API paths  *(confidence: medium)*

There is NO official Google API for the consumer product (free Colab or Colab Pro/Pro+). The consumer product is a website-only experience whose only programmatic surfaces are the Google Drive API (for CRUD on the .ipynb file) and unsupported reverse-engineered web endpoints; there is no Google-supported way to start a consumer Colab GPU runtime or run cells via API. The only OFFICIAL, supported, fully-from-code path to "run my ML notebooks on a GPU entirely from code" is Colab Enterprise (a part of Vertex AI), via the aiplatform.googleapis.com REST API resources projects.locations.notebookExecutionJobs (run a notebook headless on a GPU), notebookRuntimeTemplates, notebookRuntimes, and the Vertex AI schedules resource for recurring runs. This is a genuinely strong fit: you submit an .ipynb (inline, GCS, or Dataform), pick a machineSpec with an acceleratorType (T4/L4/A100/A100-80GB/H100/H200/B200, etc.), and outputs are written to GCS — no website needed. Vertex AI Workbench Instances is a second official path but is VM/JupyterLab-instance oriented (create/start/stop a GPU VM, then drive Jupyter), better for interactive/long-lived work than fire-and-forget execution. Both use standard Google Cloud auth (ADC / service accounts / OAuth2 user creds with the cloud-platform scope) and standard IAM. The big tradeoffs vs consumer Colab Pro: Colab Enterprise has NO subscription/CU model and NO free GPU tier — you pay raw Compute Engine VM + GPU rates per hour and must have GPU quota in your project, so it is "real cloud billing" not the $10-50/mo fixed-cost Colab Pro experience. For an agent/developer that needs deterministic, headless, scriptable GPU notebook execution, Colab Enterprise notebookExecutionJobs is the right official primitive; for the consumer Colab Pro cost profile there is no official API at all.

## Alternative GPU-notebook platforms  *(confidence: medium)*

I evaluated nine GPU notebook/compute platforms for use as fallbacks behind a Colab-control package, scored on API maturity, GPU access, auth model, and how cleanly each fits a common "remote execution backend" interface (submit code/notebook -> run on GPU -> poll status -> stream/fetch logs -> retrieve outputs). The landscape splits into three tiers. (1) Best fits behind Colab: Modal (Python-native, sandboxes for arbitrary code, per-second billing, mature SDK), Hugging Face Jobs (the closest 1:1 to "run arbitrary script/Docker image on a GPU flavor" with run_job/run_uv_job, pay-as-you-go), and Kaggle Notebooks (the only true free-GPU Colab analog with a real push/pull kernels API, but fragile and quota-limited). (2) Solid IaaS-grade backends needing more glue: RunPod (REST API GA March 2025 + Python SDK), vast.ai (mature CLI/SDK marketplace), Lightning AI (Studio/Job SDK), Vertex AI CustomJob, and SageMaker (boto3/SDK training+processing jobs). (3) Avoid: Paperspace Gradient — its programmatic Gradient AND Core APIs were deprecated 15 July 2024 and notebook/machine endpoints no longer work, so it is not a viable programmatic backend today. My recommended fallbacks behind Colab are Modal (primary, for arbitrary code + GPUs + best ergonomics), Hugging Face Jobs (secondary, closest semantic match + cheap), and Kaggle Notebooks (the free-tier analog when cost matters more than reliability). Important caveat: Colab itself has no official public API, so any "control Colab without the website" effort is inherently fragile; these alternatives are where a provider-abstraction layer delivers real, durable value.

Full CRUD on Pods, Serverless Endpoints, Network Volumes, Container Registry auth. ### Candidate approaches surfaced
- **Modal as the primary code-execution backend (Sandbox + Function)** — Wrap modal.Sandbox.create() for arbitrary/agent-generated code and modal.Function (gpu=...) for structured workloads. An external process authenticates with a token id/secret, calls modal.App.lookup(create_if_missing=True), spawns a GPU sandbox, runs commands via sb.exec(), mounts modal.Volume for file I/O, and reads stdout/stderr streams. Deployed functions are also callable from outside via modal.Function.from_name(). Map the package's submit/status/logs/fetch interface onto sandbox lifecycle + volumes.
- **Hugging Face Jobs as the 'run any script/Docker image on a GPU flavor' backend** — Use huggingface_hub.run_job(image, command, flavor, env, secrets, volumes, timeout) or run_uv_job() for inline-dependency scripts. Jobs run in the background returning JobInfo(id,url); the package polls inspect_job() for COMPLETED/ERROR, streams fetch_job_logs(), reads fetch_job_metrics(), and cancels via cancel_job(). Mount HF model/dataset repos or storage buckets for inputs/outputs. Auth is the standard HF token. This maps almost 1:1 onto a generic submit/poll/log/fetch backend interface.
- **Kaggle Notebooks as the free-tier Colab analog (kernels push/pull)** — Generate kernel-metadata.json (kernel_type 'notebook' with an .ipynb, or 'script'; enable_gpu 'true', enable_internet, dataset/model sources), run `kaggle kernels push` (which uploads AND triggers a headless GPU run), poll `kaggle kernels status` until complete, then `kaggle kernels output` to download artifacts. Wrap the kaggle-api KaggleApi methods (kernels_push, kernel_status, kernel_output) behind the package's backend interface. Auth via kaggle.json API token.
- **Marketplace-IaaS adapter (RunPod / vast.ai) behind a provision-then-exec shim** — Provider adapter calls RunPod REST (POST /v1/pods) or vast.ai SDK (search_offers -> create_instance) to provision a GPU pod/VM with a chosen image, waits for ready, then transfers code and runs it over SSH/exec, streams logs, pulls outputs, and terminates. Normalizes both behind the common submit/status/logs/fetch interface as 'ephemeral machine' backends.
- **Hyperscaler managed-job adapter (Vertex AI CustomJob / SageMaker) for enterprise-durable fallback** — Adapter builds/pushes a container (Artifact Registry / ECR) or uses prebuilt images, then submits a Vertex CustomJob (machine_type + accelerator_type/count, staging bucket) or a SageMaker training/processing job (boto3 create_training_job, instance ml.g5/p4d/p5, S3 channels). Polls job state, streams logs (Cloud Logging / CloudWatch), and reads outputs from GCS/S3. Exposed as a high-durability, enterprise-auth backend.

### Risks
- Colab itself exposes NO official public API, so the parent project's premise (control Colab without the website) is inherently fragile and likely to break as Google changes internals; the strongest durable value of this track is the provider-abstraction/fallback layer, which should be positioned as a first-class feature, not just a backup.
- GPU pricing and per-second rates cited (Modal H100 ~$3.95/hr, RunPod, vast.ai) are point-in-time (2025-2026) and fluctuate frequently — especially vast.ai marketplace rates that move with supply/demand. Treat all pricing as indicative and re-verify at integration time.
- Kaggle quotas, session caps, and accelerator types change without notice and are enforced per-account (phone verification, fair-use throttling). An automated layer that hammers Kaggle risks throttling or ToS violations; budget for graceful degradation and backoff.
- Kaggle's lack of real-time log streaming (poll-then-fetch only) means any 'live progress' UX over Kaggle is impossible — the common backend interface must tolerate log-on-completion backends, or feature-detect streaming support.
- HF Jobs is paywalled (Pro/Team/Enterprise) and has a 30-min default timeout plus an a100-large GPU ceiling; do not assume free access or large-model capacity, and always set explicit timeouts to avoid silent kills.
- Paperspace Gradient/Core programmatic APIs are deprecated (15 Jul 2024); any code or tutorial assuming a working Gradient SDK is outdated. Do not build a Paperspace backend; the DigitalOcean 'new API' surface for notebooks/machines is not clearly documented enough to rely on.
- Marketplace IaaS (RunPod/vast.ai) requires building SSH/exec/file-transfer and teardown logic yourself; bugs here cause orphaned billable GPU instances (real cost leakage) if termination isn't guaranteed on failure paths.
- Vertex AI and SageMaker GPU access is quota-gated per region/account; a fallback that 'just works' can fail at runtime with quota-exceeded errors unless quotas are pre-provisioned.
- Several details were sourced from secondary aggregators (pricing blogs, DeepWiki) rather than the vendor's primary API reference; the deepest API-shape claims (Modal sandbox, HF Jobs, Kaggle metadata) are from primary docs, but exact GPU SKUs/prices should be re-confirmed against official pricing pages before shipping.

## Browser-automation feasibility  *(confidence: medium)*

Browser automation is a viable but high-maintenance transport for driving consumer Colab Pro headlessly, and it is the ONLY transport that does not require the website in the sense of "no human clicking" — it still fundamentally depends on rendering Colab's real web app. The hard problem is not Colab itself but Google account authentication: Google explicitly blocks sign-in from automation-controlled and headless browsers (official Google Developers Blog), so you cannot reliably script the username/password/2FA flow. The practical and essentially only durable answer is to authenticate ONCE interactively (a real human, in a headed browser) into a persistent browser profile / storageState, then reuse that authenticated state for all subsequent headless runs. For driving Colab, the robust layer is the DOM/UI (open notebook -> Runtime > Run all / Ctrl+F9 -> poll cell-output DOM for results and completion), optionally tracing the kernel WebSocket read-only via Playwright's page.on("websocket") for progress. Trying to intercept and *drive* the kernel WebSocket protocol from inside the page is fragile: Colab proxies the kernel over a socket.io tunnel (/tun/m/<id>/socket.io) guarded by an x-colab-tunnel: Google header, and google.colab.kernel is only exposed inside per-cell output iframes (sandboxed), not the top-level page namespace, so re-implementing execute_request from page.evaluate is impractical. Playwright is the better choice over Selenium (first-class persistent context, storageState including IndexedDB, native WebSocket frame observation, auto-waiting). Realistic failure modes: Google re-auth/suspicious-login/reCAPTCHA challenges that invalidate the stored session, idle/12h disconnects and "are you a robot?" prompts, Colab DOM churn breaking selectors, and the perpetual cat-and-mouse of headless/CDP bot detection. Confidence is medium because Colab internals are undocumented and change frequently.

### Key findings
- Google OFFICIALLY blocks automated and headless browser sign-in. Per Google's Developers Blog, a browser is disallowed if it 'provides automation features... including scripts that automate keystrokes or clicks, especially to perform automatic sign-ins', if it is a 'headless browser', or if it spoofs another browser's User-Agent or proxies/alters network traffic. This is anti-MITM policy, not a bug to be patched — scripting the Google login form directly is a dead end.
- Therefore the practical auth model is: log in ONCE interactively as a human in a headed real browser, capture the authenticated state, and reuse it headlessly. Two mechanisms: (a) Playwright launch_persistent_context(user_data_dir=...) which persists the full Chrome profile (cookies, localStorage, IndexedDB, cache) on disk like a normal browser; (b) context.storage_state(path=...) + new_context(storage_state=...). Current Playwright docs state storage_state now covers cookies, localStorage AND IndexedDB, but NOT sessionStorage (needs add_init_script workaround). Google auth tokens partly live in IndexedDB, so persistent_context is the safer choice.
- Even with reused state, the session is not permanent. Google periodically forces re-authentication and fires suspicious-login / reCAPTCHA / device-verification challenges, especially when the same account is driven from a headless/datacenter context. When that fires, automation cannot solve it — a human must re-seed the profile. Community reports confirm Google blocks Selenium/undetected-chromedriver Google logins 'after a few days' of automated activity. Advice from practitioners: authenticate as rarely as possible to keep the account 'cold'.
- Colab UI automation is the robust transport. Open https://colab.research.google.com/drive/<id> (or github/ URL), wait for the kernel to connect, trigger Runtime > Run all (keyboard Ctrl+F9 / Cmd+F9), then poll the cell-output DOM for results and the run-state/spinner to detect completion. This is selector-fragile but does not require understanding Colab's private protocol.
- Driving the kernel WebSocket directly from inside the page is HARD and fragile. DagsHub's 'Reverse Engineering Google Colab' shows the kernel is reached via a reverse-proxy tunnel at /tun/m/<id>/socket.io (socket.io, not raw Jupyter v1.kernel.websocket.jupyter.org), and Colab 'refuses to proxy any requests unless they have the x-colab-tunnel: Google header set', and the official proxy is GET-only (view, not run). Replaying this externally is blocked; doing it in-page is complicated by sandboxing (next finding).
- google.colab.kernel (the JS that invokes the Python kernel) is exposed ONLY inside per-cell output iframes, which are sandboxed, NOT in the top-level page namespace (Matteo Ferla blog). So you cannot simply page.evaluate() against google.colab.kernel.invokeFunction from the top frame to push execute_request. You would have to operate within an output iframe's context, which Colab controls and changes. This makes 'intercept the kernel WS and drive it from CDP' far less attractive than it sounds.
- Playwright CAN observe the kernel WebSocket read-only for free: page.on('websocket') then ws.on('framereceived')/ws.on('framesent') with payload as str|bytes. This is useful for detecting execution progress/idle status and streaming outputs, but Playwright's public API only OBSERVES frames; WebSocketRoute (page.route_web_socket) is the documented path to intercept/modify, and it is primarily designed for mocking, not for re-driving a socket.io-multiplexed kernel protocol.
- Bot/headless detection is a real, escalating risk. Per Castle.io (2025), since Nov 2022 Chrome unified headless+headful codebases so classic headless fingerprinting is 'largely ineffective', but CDP-presence detection is now the strong signal: enabling the CDP Runtime domain causes observable serialization side-effects (e.g. Runtime.consoleAPICalled on logging an error with a custom getter) that pure-CDP tools (Playwright, Selenium) expose. navigator.webdriver and stealth-plugin patches are easily detected. Newer CDP-minimal frameworks (nodriver, selenium-driverless) avoid CDP via OS-level input, but Google-grade detection success is unproven.
- Playwright is the better tool than Selenium here: native persistent context + storageState, IndexedDB in storage state, built-in WebSocket frame events, auto-waiting (reduces flaky run-all polling), and a single CDP connection. Selenium's main edge is the mature undetected-chromedriver ecosystem, but undetected-chromedriver is itself open-source and thus signature-detectable, and is reported to be detected by Google login. NOTE: google-colab-selenium (jpjacobpadilla, updated May 2026) runs Selenium INSIDE Colab to control other sites — it is the inverse of this task and not directly reusable.
- Stability/reconnection burden is significant. Colab disconnects on ~90-min idle and a hard ~12h cap; long runs trigger 'are you still there / I'm not a robot' prompts that scripts cannot solve. You must implement: keep-alive interaction, reconnection (re-open notebook, re-attach to the still-running runtime, re-bind output observers), and checkpointing assumptions. The whole stack breaks whenever Colab ships front-end DOM changes or Google tightens login challenges, so expect ongoing maintenance rather than a build-once solution.

### Candidate approaches surfaced
- **Playwright + persistent context, human-seeded auth, DOM-level Colab control (recommended)** — Operator logs into Google once in a headed real Chrome with a Playwright launch_persistent_context(user_data_dir). That on-disk profile (cookies + localStorage + IndexedDB) is reused for all later headless runs. Automation navigates to the notebook URL, waits for kernel connect, fires Run all (Ctrl/Cmd+F9 or the Runtime menu), and reads results by polling the cell-output DOM and run-state indicators. File transfer is done via Colab's UI file panel or, more robustly, by having notebook cells push/pull to Google Drive / GCS.
- **Persistent profile + read-only WebSocket trace for orchestration** — Same auth/profile model as above, but instead of scraping outputs purely from the DOM, attach page.on('websocket') and ws.on('framereceived') to the kernel socket to detect kernel idle/busy status, execution counts, and streamed stdout, using that as the signal for 'cell finished' and to capture textual outputs. Still trigger execution via the UI (Run all), never by injecting into the socket.
- **Selenium + undetected-chromedriver with reused user-data-dir** — Use Selenium with undetected-chromedriver, a persistent Chrome user-data-dir seeded by a manual login, then drive the Colab UI similarly to the Playwright approach. Relies on undetected-chromedriver's patches to reduce navigator.webdriver/CDP signals.
- **In-page kernel driving via output-iframe context (NOT recommended)** — Attempt to execute code by reaching google.colab.kernel.invokeFunction / the kernel WS from JavaScript injected into a Colab cell-output iframe, effectively reusing Colab's own auth/tunnel because the calls originate same-origin from the page Colab itself controls.

### Risks
- Google can break the whole approach at the auth layer at any time. The 'block automated/headless sign-in' policy is active and tightening (LSA deprecation completed 2025), and Google can invalidate stored sessions or raise challenge frequency for any account it deems suspicious.
- Suspicious-login / reCAPTCHA / device-verification challenges require a human and cannot be solved by the automation; an unattended fleet will eventually stall on a challenge it cannot pass. Using CAPTCHA-solving services against Google login risks account suspension and likely violates Google ToS.
- Running consumer Colab via automation may violate Google/Colab Terms of Service and trigger account-level enforcement (suspension), risking the user's real Google account, not just a session. This is a policy/legal risk, not just technical.
- CDP-based detection (Playwright/Selenium) is a known, demonstrated signal (Castle.io 2025); even with stealth, Google-grade detection on accounts.google.com and possibly Colab can flag the session, and stealth patches are themselves detectable.
- Colab's front-end DOM and kernel/socket.io protocol are undocumented and change without notice; selectors and any protocol parsing will break periodically, producing a continuous maintenance tax.
- Persistent profiles and storageState files contain live session credentials — if leaked they let an attacker impersonate the user's full Google account; they must be encrypted at rest and never committed to source control (Playwright docs explicitly warn this).
- Idle (~90 min) and hard (~12 h) disconnects plus 'are you still there' prompts mean long GPU/ML jobs need robust reconnection + checkpointing; a naive run-all-and-wait design will silently lose work.
- Headless servers are typically datacenter IPs; Google treats new datacenter IP + automation as high-risk, increasing challenge rate versus a residential/home seed location.

### Open questions
- Does reusing a persistent profile from a residential-seeded login survive long-term when subsequently driven headless from a datacenter IP, or does Google force re-auth within days as some reports suggest? (Likely fragile; needs empirical testing per-account.)
- What is Colab's CURRENT (2026) kernel transport — is it still socket.io over /tun/m/<id>/, or has it moved closer to standard Jupyter v1 WebSocket? The DagsHub analysis predates 2026 and Colab changes silently. Requires live DevTools inspection to confirm.
- Can completion/output be reliably read by observing the kernel WebSocket frames given socket.io multiplexing, or is DOM polling actually more stable in practice? Needs a spike comparing both.
- How aggressively does Colab itself (as opposed to Google login) detect automation inside an authenticated session — does it ever serve its own bot challenge or rate-limit Run all? Unknown from public sources.
- Does Colab Pro specifically alter any of this (different timeouts, GPU allocation prompts, 'you cannot currently connect to a GPU' interstitials) in ways that add new UI states the automation must handle?
- Is there any officially supported path (e.g., Colab Enterprise on Vertex AI, or local-runtime jupyter_http_over_ws) that achieves the same goal without fighting consumer login detection — which would be far more durable than browser automation of consumer Colab?

## Google auth & session mechanics  *(confidence: medium)*

Programmatic Colab access splits into three transports, each needing different auth artifacts. (1) The OFFICIAL/sanctioned path uses Google OAuth2 with the restricted scope `https://www.googleapis.com/auth/colaboratory`; the resulting access token is sent as `Authorization: Bearer <token>` to Colab's backend REST API under `https://colab.research.google.com/tun/m/*` to allocate runtimes (incl. GPU/TPU), then a short-lived runtime-proxy token + WebSocket reaches the Jupyter kernel. Google's own tooling here is `googlecolab/google-colab-cli` (now the recommended tool) and the `googlecolab/colab-vscode` extension. The big catch: the `colaboratory` scope is NOT a publicly listed/grantable API scope — Google maintainers confirmed this on the colab-mcp repo, removed runtime code from colab-mcp, and have not (as of the sources reviewed) clearly shipped a public OAuth client for arbitrary external users. So external feasibility is genuinely uncertain. The practical workaround is to reuse the Colab VS Code extension's public OAuth client_id/secret via an installed-app loopback+PKCE flow (what `xeodou/colab-cli` does). (2) The REVERSE-ENGINEERED browser path uses your logged-in Google cookies (SID/HSID/SSID/APISID/SAPISID) plus a SAPISIDHASH `Authorization` header and an `x-colab-tunnel: Google` header to hit the same `/tun/m/*` proxy and the kernel socket.io. (3) The RUNTIME-SIDE `google.colab` library (auth.py, drive.py) is only relevant for code already running inside Colab; it talks to an on-VM metadata server (`TBE_CREDS_ADDR`/`TBE_EPHEM_CREDS_ADDR`) and the frontend comm channel — not usable by an external client. For Drive file transfer specifically, an external client should prefer a Google service account (scope `https://www.googleapis.com/auth/drive`) with the folder shared to the SA email — fully supported and stable, decoupled from Colab auth. Store all secrets (refresh tokens, cookies) in the OS keychain via Python `keyring` (macOS Keychain backend), keyed by account email for multi-account support.

### Key findings
- OFFICIAL OAUTH SCOPE: The Colab backend OAuth scope is `https://www.googleapis.com/auth/colaboratory`. Both `xeodou/colab-cli` and `colab-mcp --enable-runtime` request exactly this scope. The access token is attached to backend calls as `Authorization: Bearer <token>` (confirmed in colab-vscode's ColabClient).
- SCOPE IS RESTRICTED / NOT PUBLIC (central fragility): On `googlecolab/colab-mcp` Discussion #41, a user reported `googleapis.com/auth/colaboratory` is not a publicly available API scope in Google Cloud Console; maintainer 'teeler' replied 'you are correct that this is not a publicly available API' and said the team was deciding whether to 'publish our own client info so you can use it, or making the API public.' They then REMOVED runtime code from colab-mcp entirely and now point users to `googlecolab/google-colab-cli`.
- PRACTICAL EXTERNAL WORKAROUND: `xeodou/colab-cli` (community) authenticates by DEFAULT using 'the public OAuth2 credentials from Google's Colab VS Code extension' via an 'OAuth2 loopback flow with S256 PKCE' (installed-app model). Users may override with env vars `COLAB_CLIENT_ID` / `COLAB_CLIENT_SECRET`. Tokens cache at `~/.config/colab/token.json` and auto-refresh. This is the most concrete proof external programmatic access is achievable today, by borrowing the VS Code extension's published desktop OAuth client.
- SELF-REGISTERED OAUTH CLIENT IS POSSIBLE BUT LIMITED: To use your own client you create a Desktop-app OAuth client in Google Cloud Console, add the `colaboratory` scope on the consent screen, and add your account as a test user. Consequences (per setup notes): the app stays 'unverified' (scary consent warning), is capped at 100 test users, AND because it's an external app in 'Testing' status, Google issues REFRESH TOKENS THAT EXPIRE IN 7 DAYS (per Google's own OAuth2 docs) — a serious operational limitation unless the app gets verified/published.
- BACKEND REST API MAP (official transport): colab-vscode's ColabClient calls `https://colab.research.google.com` with: `GET/POST /tun/m/assign` (allocate runtime; query params `variant`=GPU|TPU and `accelerator`=T4|A100 etc.), `GET /tun/m/assignments` (list), `GET/POST /tun/m/unassign/{endpoint}` (release), `GET /tun/m/runtime-proxy-token` (returns RuntimeProxyInfo {token, url, tokenExpiresInSeconds}), `GET /tun/m/{endpoint}/api/sessions`. State-changing POSTs require a two-phase XSRF token `X-Goog-Colab-Token`.
- KERNEL CONNECTION: The kernel is reached via a tunnel proxy URL returned by `/tun/m/runtime-proxy-token` (the proxy token is SHORT-LIVED, with explicit `tokenExpiresInSeconds`). Execution is done over a Jupyter WebSocket sending `execute_request` messages; file transfer uses the Jupyter Contents API `/api/contents` (and `/files/`). The reverse-engineered path reaches the same kernel via `https://colab.research.google.com/tun/m/<id>/socket.io`.
- SAPISIDHASH (reverse-engineered cookie auth): Computed as `<TIMESTAMP_MS>_<SHA1("<TIMESTAMP_MS> <SAPISID> <origin>")>` and sent as `Authorization: SAPISIDHASH <timestamp>_<hash>`, with the origin (e.g. https://colab.research.google.com) also reflected in an Origin/X-Origin header. Requires the logged-in Google cookies SID, HSID, SSID, APISID, SAPISID. NOTE: timestamp granularity differs across sources (some Google code uses seconds, the cited gist uses ms) — verify empirically against the live frontend.
- TUNNEL HEADER: Reaching `/tun/m/<id>/...` from outside the Colab frontend requires the header `x-colab-tunnel: Google` plus the full Google cookie jar. The header exists to block requests from ordinary browser tabs (anti-XSS), so it must be set manually by any non-browser client (dagshub reverse-engineering writeup).
- RUNTIME-SIDE `google.colab` IS NOT AN EXTERNAL CLIENT API: `auth.py` `authenticate_user()` runs gcloud (`gcloud auth login --enable-gdrive-access --no-launch-browser --quiet`) or an ephemeral path via the kernel<->frontend comm (`_message.blocking_request('request_auth', {'authType':'auth_user_ephemeral'})`), requests only the `email` scope for ADC, and writes creds to the VM's `credentials.db`/`adc.json`. `drive.py` mounts Drive via an on-VM metadata server using env vars `TBE_CREDS_ADDR` / `TBE_EPHEM_CREDS_ADDR` (`--metadata_server_auth_uri=.../computeMetadata/v1`) or an interactive OAuth-code paste flow. None of this is reachable from off-VM; it only works in code already executing inside the runtime.
- DRIVE FILE TRANSFER (most stable option): For programmatic file transfer, prefer the Google Drive API directly with a SERVICE ACCOUNT (scope `https://www.googleapis.com/auth/drive`, `from google.oauth2 import service_account`). Share the target Drive folder with the service account's email and grant Editor. This is fully documented, stable, and entirely independent of Colab's fragile auth — decoupling file transfer from runtime control is a strong architecture choice.
- TOKEN LIFETIMES: Google OAuth2 access tokens live ~1 hour (3600s) and are auto-refreshed by `google-auth` using the refresh token. Refresh tokens expire if: unused 6 months; the consent screen is in 'Testing' status (7 days); password change with Gmail scopes; user revocation; exceeding 100 live refresh tokens per (account, client_id); or admin/session policies. Colab runtime-proxy tokens are separately short-lived (TTL given by `tokenExpiresInSeconds`) and must be re-fetched.
- SECRET STORAGE: Python `keyring` is the right local store; on macOS it uses the Keychain backend (needs macOS 11+, Python 3.8.7+ universal2). Pattern: `keyring.set_password(service='your-app', username='<account-email>', password=<refresh_token_or_blob>)`. Use one service name + per-account-email usernames to support MULTIPLE Google accounts cleanly. This avoids plaintext token.json files; note tools like xeodou/colab-cli currently use a plaintext `~/.config/colab/token.json` instead, which is a weakness to improve on.
- macOS BROWSER-COOKIE EXTRACTION CAVEAT: `browser_cookie3` can read cookies from Chrome/Safari/Firefox/Brave/Edge etc. On macOS, Chromium cookie encryption uses the system Keychain (so extraction generally works with a Keychain unlock prompt). Chrome 127's App-Bound Encryption (July 2024) that broke `browser_cookie3` is WINDOWS-ONLY today (DPAPI replacement) — but Google has stated intent to expand it, so the Windows cookie-extraction path is already broken/fragile and macOS could follow. Treat cookie extraction as the least durable approach.

### Candidate approaches surfaced
- **A. OAuth2 installed-app flow borrowing the Colab VS Code extension's public client (RECOMMENDED primary)** — Use Google's installed-app/loopback OAuth2 flow with S256 PKCE, requesting scope `https://www.googleapis.com/auth/colaboratory`, using the client_id/client_secret published in the public `googlecolab/colab-vscode` extension (the approach `xeodou/colab-cli` ships). Persist the resulting refresh token in the OS keychain. Use the access token as `Authorization: Bearer` against `https://colab.research.google.com/tun/m/*` to assign a runtime, fetch a runtime-proxy token, and open the Jupyter WebSocket. Refresh access tokens via google-auth; re-fetch the short-lived proxy token on expiry.
- **B. Self-registered Desktop OAuth client with the colaboratory scope** — User creates their own Desktop-type OAuth client in Google Cloud Console, manually adds the `colaboratory` scope to the consent screen, adds themselves as a test user, and supplies client_id/secret (e.g., via COLAB_CLIENT_ID/SECRET or `--client-oauth-config`). Same backend flow as Approach A thereafter.
- **C. Reverse-engineered cookie + SAPISIDHASH transport** — Extract the logged-in Google cookies (SID/HSID/SSID/APISID/SAPISID) from the user's browser (e.g., via browser_cookie3), compute `SAPISIDHASH = <ts>_<SHA1("<ts> <SAPISID> <origin>")>`, and call the same `/tun/m/*` endpoints and `/tun/m/<id>/socket.io` kernel socket with `Authorization: SAPISIDHASH ...`, the proper Origin/X-Origin, and `x-colab-tunnel: Google`.
- **D. Service account for Drive only, OAuth for compute** — Decouple concerns: use a Google service account (scope `auth/drive`) with target folders shared to the SA email for ALL file transfer, and use Approach A/B purely for runtime/kernel control. Files land in Drive; the Colab runtime reads/writes the same Drive.
- **E. Browser automation (Playwright/CDP) driving the real Colab frontend** — Automate a real logged-in browser (persistent profile) to open notebooks and execute cells, letting the frontend handle all SAPISIDHASH/cookie/comm auth internally. The external client only drives the UI / reads kernel output via CDP.

### Risks
- The `colaboratory` OAuth scope is restricted and not publicly grantable; Google explicitly said so and is undecided on publishing a public client or opening the API. The entire official-transport feasibility for external users rests on either borrowing the VS Code extension's public client (could be locked down at any time) or self-registration (may not be allowed to add the scope at all).
- Self-registered external apps in 'Testing' status get refresh tokens that expire after 7 DAYS — any product built on Approach B will silently require weekly re-consent unless it goes through Google app verification for a niche scope.
- All `/tun/m/*` backend paths, the XSRF `X-Goog-Colab-Token` flow, runtime-proxy token shape, and the `x-colab-tunnel: Google` header are undocumented internals reverse-engineered from colab-vscode/dagshub; Google can change them without notice and Colab internals are known to churn.
- SAPISIDHASH details (timestamp seconds vs ms, exact hash input string) are inconsistent across sources and over time; the cookie path needs continuous empirical re-verification.
- Browser cookie extraction is actively being broken by Chrome App-Bound Encryption (Windows now; macOS/passwords/tokens are stated future targets). Any cookie-based approach is on borrowed time and is a severe credential-theft-grade liability if mishandled.
- Reverse-engineered and browser-automation approaches likely violate Colab Terms of Service and risk account suspension; this is a product/legal risk, not just technical.
- Storing full Google cookies or refresh tokens for the powerful `colaboratory`/`drive` scopes is high-value secret material; a leak compromises the user's Google account. OS keychain mitigates at-rest risk but process-memory and exfil risks remain.
- Multi-account support multiplies the 100-refresh-tokens-per-(account, client_id) limit risk if the same borrowed client_id is reused across many users/accounts — older tokens can be silently invalidated.
- google-colab-cli's README (the now-recommended official tool) does not clearly document whether it ships a usable public client for external users; the auth feasibility for non-Google users is therefore unverified from the docs reviewed and should be tested empirically before committing an architecture.

### Open questions
- Does `googlecolab/google-colab-cli` ship a built-in/public OAuth client that lets arbitrary external (non-Google) users authenticate to the `colaboratory` scope out of the box, or must they bring their own client? The README reviewed did not say. This is THE load-bearing question and needs hands-on testing.
- Can an arbitrary Google Cloud project actually ADD `https://www.googleapis.com/auth/colaboratory` to its consent screen, or is the scope gated server-side to first-party clients? (Determines whether Approach B is even possible.)
- What is the exact, current SAPISIDHASH input format the live Colab frontend uses (seconds vs ms, single vs triple hash SAPISIDHASH/SAPISID1PHASH/SAPISID3PHASH)? Needs capture from a live session.
- Exact request/response schemas for `/tun/m/assign`, the full set of accelerator enum values currently accepted, and the precise WebSocket URL + auth params derived from `runtime-proxy-token` — only partially captured from colab-vscode docs.
- Has Google's stance from colab-mcp Discussion #41 advanced since (did they publish client info or make the API public)? The thread was closed by redirecting to google-colab-cli without a definitive public-client answer.
- Whether borrowing the colab-vscode public client_id for a separate product violates Colab/Google API ToS or the extension's license, and how durable that client_id is.
- For the runtime-proxy WebSocket, is the proxy token passed as a query param, a subprotocol, or a header, and how exactly is the kernel session bound to it? (TTL is known; transport detail is not.)
- Does macOS Chromium cookie extraction still work reliably in 2026 given Google's stated intent to extend App-Bound-style protection beyond Windows?
